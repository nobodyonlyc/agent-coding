# Evidence: Multi-agent projection in install.sh

## What was built
Replaced the F01 skeleton with real, idempotent projection:
- Claude Code: symlink .claude/skills -> skills-src/skills (relative, safe if a real dir exists).
- Antigravity: generate .agent/workflows/harness-<name>.md adapters for each workflow-* (description
  copied from frontmatter, pointing to the canonical SKILL.md).
- Codex: path-based note (no projection).
- wire_hooks(): merges config-templates/claude.settings.json via jq if present, else instructs;
  skips cleanly when templates are absent (F12 provides them).

## Test
F11 verification ran and passed, plus a real execution:
```
grep -q .claude/skills install.sh && grep -q .agent/workflows install.sh && validate -> PASS
bash install.sh -> symlinked .claude/skills (35 skills), generated 5 antigravity adapters, codex note, hooks skipped
bash -n install.sh -> syntax OK
```
Confirmed .claude/skills resolves to 35 skills and a sample adapter (harness-workflow-intake.md) is correct.

## Review
Self-review: idempotent (rm -f symlink before relink; never deletes a real dir); only workflow-*
get adapters (leaf skills are orchestrator-invoked, not /harness-* entry points) — matches CATEGORIES.
Relative symlink via os.path.relpath is portable. wire_hooks degrades gracefully without jq/templates.
Generated .claude/skills + .agent/workflows are projection artifacts (a consuming project may gitignore them).
No issues.

## Key decisions made
- Only workflows get Antigravity /harness-* adapters; leaf skills stay path/symlink-accessible but
  are invoked by the orchestrator, keeping the command surface small.
- Projection is regenerated by install.sh after each submodule update, so one update propagates to all agents.
