1 · Executive summary

This is a mature, well-architected agent harness, not a prompt collection. Its defining idea is a clean separation between a thin orchestration spine (workflow / plan / dev / test / check / ship skills that own what to do and the gates) and 36 vendored MIT expert skills that supply per-language / per-framework craft. Process discipline is enforced mechanically through three layers: an SQLite-backed backlog (WIP=1, immutable verification commands), proximity docs (per-feature plan.md + evidence.md), and a 16-hook gate chain wired into the agent runtime (PreToolUse/Stop/SessionStart).

All static-integrity and behavioral tests passed: the validator is green, every skill has valid frontmatter, all 65 resource links resolve, the hook wiring is exactly consistent (16 wired = 16 on disk, zero orphans/dangling), and the gate chain blocks every malformed fixture while allowing every well-formed one and every negative control. The main deductions are operational: the harness binary is not in the repo (the whole flow is inert until installed), a stale duplicate hooks/ tree at repo root invites drift, and the evidence gates verify the record rather than re-executing tests — a deliberate boundary that should be understood.

2 · Architecture overview

Layers

  • Harness CLI — backlog state (SQLite ↔ features.json), WIP=1, verify runs immutable commands, session lifecycle.
  • Orchestration spine — workflow-* (5), plan-* (4), dev-* (9), test-* (7), check-* (7), design-* (6), ship-* (4), core-* (4), opt-*.
  • Expert bridge — expert-skills-map.md delegates craft to 36 vendored experts (python-pro, react-expert, playwright-expert, test-master, …).
  • Resources — shared conventions (persona-mode, autonomy-mode, token-budget, step-gate, task-state) + language convention files.
  • Gate layer — 16 hooks enforcing phase, quality, review, test rigor, persona, context budget, git safety.
  • Token optimization — caveman nested submodule (v1.9.0) as a standing rule.

Lifecycle (enforced)

resume → start(plan-gated) → work
   → unit → fix → repeat
   → verify(evidence-gated) → stop

WIP = 1            (single in_progress)
verifications      frozen at start (anti-bias)
evidence.md        required for verify
review + test      hard gates before passing
The gate chain makes these rules non-bypassable from inside the agent loop: start is blocked by an unfilled plan; verify is blocked by missing/sparse evidence, a non-green test verdict, a missing IT test-case table, or an uncovered UI flow.

3 · Methodology

Two complementary passes, all read-only / sandboxed:

  • Static integrity — run scripts/validate.sh and lint-expert-map.sh; verify every SKILL.md frontmatter; resolve all ../X/SKILL.md and ../../resources/*.md links from each skill's own directory; reconcile hooks wired in settings.json against hooks on disk; bash -n all hooks.
  • Behavioral gate testing — drive each verify-time gate with crafted evidence.md fixtures (one well-formed, several malformed) plus negative-control commands, asserting allow(0)/block(1); then a regression sweep running the new always-on gate against all 45 existing features to detect false positives.

7 · Strengths

  • Mechanically enforced process. WIP=1, plan-before-start, evidence-before-verify, and review/test hard gates are implemented as runtime hooks — not advice an agent can quietly skip.
  • Anti-self-evaluation design. Verification commands are frozen at start; evidence requires both a ## Test and an independent ## Review section, mitigating "grade your own homework" bias.
  • Spine + expert bridge. The orchestration skills stay thin and stack-agnostic while 36 MIT experts carry idioms — easy to add a language without touching the workflow logic.
  • Test rigor is now first-class. IT requires concrete test-case tables (steps + expected output); UI E2E requires a full-coverage matrix; non-green verdicts block verify (feature F45).
  • Clean engineering hygiene. Zero broken links, exact hook wiring, idempotent installer, token-budget discipline via caveman.

9 · Recommendations (prioritized)

  • P1 — Make the loop reproducible. partly done Binary now installed from GitHub Releases. Remaining: have init.sh auto-invoke install-harness.sh (or fail loudly) when ./harness is missing, so a fresh clone runs start/verify out of the box.
  • P1 — Remove the root hooks/ tree. done Deleted; README updated to point at .harness/skills-src/hooks/ as the single source of truth.
  • P2 — Add a CI job running validate.sh, the lints, and a checked-in version of the gate fixture matrix on every push to harness-skills.
  • P2 — Tie the gates to selection, not just the record. Have check-test-strategy emit the selected tiers into the feature/evidence machine-readably, then let the gate assert that every selected tier appears and passes (closes F4: enforce "unit required" for code tasks).
  • P3 — Optionally cross-check execution. Where cheap, have harness verify stamp a signed/structured test result into evidence that the gate reads, narrowing the fabrication gap (F3).
  • P3 — Centralize gate-format docs into one canonical reference linked by both skills and hooks.
